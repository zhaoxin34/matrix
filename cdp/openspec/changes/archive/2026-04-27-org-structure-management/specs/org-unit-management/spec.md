## 新增需求

### 需求：创建组织单元
系统**必须**允许授权用户通过填写名称、编码（唯一）、类型（company/branch/department/sub_department）、上级组织以及可选的负责人来创建组织单元。

#### 场景：成功创建部门
- **当** 管理员提交包含名称、唯一编码、类型和 parent_id 的合法创建请求时
- **则** 系统依据上级层级计算 level，创建该组织单元，更新 org_unit_closure 表，并返回新建单元的数据

#### 场景：编码重复时拒绝创建
- **当** 管理员提交的创建请求中包含已存在的编码时
- **则** 系统返回 400 错误，提示编码必须唯一

### 需求：编辑组织单元
系统**必须**允许授权用户更新现有组织单元的名称、编码、类型、负责人、排序值和状态。

#### 场景：成功修改组织名称
- **当** 管理员对已存在的组织单元提交合法的更新请求时
- **则** 系统持久化变更并返回更新后的单元数据

#### 场景：编辑不存在的单元
- **当** 管理员对不存在的单元 ID 提交更新请求时
- **则** 系统返回 404 错误

### 需求：删除组织单元
系统**必须**仅允许删除没有下级组织且没有员工归属的组织单元。

#### 场景：成功删除空组织
- **当** 管理员请求删除一个没有子单元且没有员工的组织单元时
- **则** 系统删除该单元，并清除闭包表中所有相关记录

#### 场景：拒绝删除非空组织
- **当** 管理员请求删除一个存在子单元或已归属员工的组织单元时
- **则** 系统返回 400 错误，提示该单元存在依赖项，无法删除

### 需求：启用与禁用组织单元
系统**必须**允许授权用户在不删除数据的情况下，将组织单元的状态在启用（active）和禁用（inactive）之间切换。

#### 场景：禁用启用中的组织
- **当** 管理员禁用一个处于启用状态的组织单元时
- **则** 该单元状态设为 inactive，并从正常树形列表中排除

#### 场景：重新启用禁用中的组织
- **当** 管理员启用一个处于禁用状态的组织单元时
- **则** 该单元状态设为 active，并重新出现在树形列表中

### 需求：获取组织树
系统**必须**以层级结构返回完整的组织树，每个节点包含 id、名称、编码、类型、层级、状态、排序值、负责人和员工数量。

#### 场景：获取完整组织树
- **当** 用户请求 GET /api/v1/org/units 时
- **则** 系统返回所有启用状态的组织单元组成的嵌套树，同级节点按 sort_order 排序

#### 场景：获取单个节点详情
- **当** 用户请求 GET /api/v1/org/units/{id} 时
- **则** 系统返回该单元的完整信息，包括其直接子节点列表

### 需求：移动组织单元
系统**必须**允许授权用户将组织单元移动到不同的上级节点，并相应更新该单元及其所有后代的层级和闭包表关系。

#### 场景：成功移动组织单元
- **当** 管理员向 PATCH /api/v1/org/units/{id}/move 发送包含合法 new parent_id 的请求时
- **则** 系统更新该单元的 parent_id 和 level，重建该单元及其所有后代的闭包表条目，并返回更新后的单元数据

#### 场景：拒绝循环移动
- **当** 管理员尝试将某个单元移动到其自身的后代节点下时
- **则** 系统返回 400 错误，提示该移动会造成循环引用
