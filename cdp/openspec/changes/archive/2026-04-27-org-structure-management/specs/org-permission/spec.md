## 新增需求

### 需求：基于角色的权限范围
系统**必须**执行四种角色的权限范围：系统管理员（全部单元）、分公司管理员（本公司及后代）、部门管理员（本部门及后代）、普通员工（所在部门及后代，仅查看）。

#### 场景：系统管理员访问全部单元
- **当** 角色为 system_admin 的用户请求组织树或员工列表时
- **则** 系统不加限制地返回全部组织单元和员工数据

#### 场景：分公司管理员仅限本公司范围
- **当** 角色为 branch_admin 的用户请求组织树时
- **则** 系统仅返回其所属分公司及其后代单元的数据

#### 场景：部门管理员仅限本部门范围
- **当** 角色为 department_admin 的用户请求员工列表时
- **则** 系统仅返回归属于其部门及后代单元的员工

### 需求：读取权限执行
系统**必须**将所有读取操作（组织树、单元详情、员工列表、员工详情）限制在请求用户的权限范围内。

#### 场景：普通员工无法查看权限外的单元
- **当** 角色为 employee 的用户请求其权限范围外的组织单元详情时
- **则** 系统返回 403 Forbidden 错误

#### 场景：有权限的用户可查看自身范围内的数据
- **当** 用户请求其权限范围内的组织单元或员工时
- **则** 系统返回请求的数据

### 需求：写入权限执行
系统**必须**将所有创建、更新、删除和调动操作限制为在其权限范围内操作的管理员角色（system_admin、branch_admin 或 department_admin）。

#### 场景：非管理员无法创建组织单元
- **当** 角色为 employee 的用户尝试创建、编辑或删除组织单元时
- **则** 系统返回 403 Forbidden 错误

#### 场景：分公司管理员无法修改权限外的单元
- **当** 角色为 branch_admin 的用户尝试编辑其所属分公司子树之外的单元时
- **则** 系统返回 403 Forbidden 错误

### 需求：基于闭包表的权限解析
系统**必须**使用 org_unit_closure 表高效解析所有祖先和后代关系，以评估用户的权限范围。

#### 场景：解析所有后代以进行权限校验
- **当** 系统评估用户是否有权访问某个组织单元时
- **则** 查询 org_unit_closure 中 ancestor_id = 用户根单元 ID 的所有记录，获取全部允许访问的 unit_id 集合，再检查目标 unit_id 是否在该集合内
