# 组织架构管理 - 产品需求文档

**版本**: v1.0
**日期**: 2026-04-21
**状态**: 草稿

---

## 1. 概述

### 1.1 产品背景

为中型企业（200-2000人）提供完整的组织架构管理能力，支持树形组织结构、员工管理、部门调动、权限控制和与 OA/审批系统的集成。

### 1.2 核心价值

- 建立统一的组织架构数据源，消除信息孤岛
- 支持复杂的组织架构（总公司 -> 分公司 -> 一级部门 -> 二级部门）
- 员工可归属任意层级组织，支持主部门 + 辅部门
- 与 OA/审批系统联动，基于组织架构确定审批链

## 2. 组织架构模型

### 2.1 数据模型

```
组织单元 (OrganizationUnit)
├── id: bigint, PK
├── name: varchar(100)          # 组织名称
├── code: varchar(50)           # 组织编码（唯一）
├── type: enum                 # 组织类型
│   ├── company                # 总公司
│   ├── branch                  # 分公司
│   ├── department              # 一级部门
│   └── sub_department          # 二级部门
├── parent_id: bigint, FK       # 上级组织（NULL 表示根节点）
├── level: int                  # 层级深度 (0=集团, 1=分公司, 2=一级部门, 3=二级部门)
├── status: enum                # 状态 (active/inactive)
├── sort_order: int             # 同级排序
├── leader_id: bigint, FK       # 负责人（员工ID）
├── created_at: datetime
└── updated_at: datetime
```

```
员工 (Employee)
├── id: bigint, PK
├── employee_no: varchar(50)    # 员工工号（唯一）
├── name: varchar(100)         # 姓名
├── phone: varchar(20)          # 手机号
├── email: varchar(100)         # 邮箱
├── position: varchar(100)      # 岗位
├── primary_unit_id: bigint, FK # 主属部门
├── status: enum                # 状态 (onboarding/on_job/offboarding)
├── entry_date: date           # 入职日期
├── dimission_date: date        # 离职日期（NULL 表示未离职）
├── created_at: datetime
└── updated_at: datetime
```

```
员工辅属部门 (EmployeeSecondaryUnit)
├── id: bigint, PK
├── employee_id: bigint, FK
├── unit_id: bigint, FK
└── created_at: datetime
```

```
员工调动历史 (EmployeeTransfer)
├── id: bigint, PK
├── employee_id: bigint, FK
├── from_unit_id: bigint, FK    # 调出部门
├── to_unit_id: bigint, FK      # 调入部门
├── transfer_type: enum         # 调动类型 (promotion/demotion/transfer)
├── effective_date: date        # 生效日期
├── reason: varchar(500)        # 调动原因
└── created_at: datetime
```

### 2.2 组织层级示例

```
集团 (level=0)
├── 北京分公司 (level=1)
│   ├── 销售部 (level=2)
│   │   ├── 华北销售组 (level=3)
│   │   └── 华南销售组 (level=3)
│   └── 市场部 (level=2)
│       ├── 品牌组 (level=3)
│       └── 活动组 (level=3)
└── 上海分公司 (level=1)
    ├── 销售部 (level=2)
    └── 客服部 (level=2)
```

## 3. 功能模块

### 3.1 组织管理 (P0)

#### 3.1.1 组织单元 CRUD

| 功能 | 描述 |
|------|------|
| 新增组织 | 选择上级组织，填写名称、编码、类型、负责人 |
| 编辑组织 | 修改组织信息，支持批量编辑 |
| 删除组织 | 校验是否有下级组织和员工，仅删除空组织 |
| 禁用/启用 | 禁用后组织不显示但数据保留 |

#### 3.1.2 组织操作

| 功能 | 描述 |
|------|------|
| 拖拽调整 | 拖拽组织节点到目标位置，改变父子关系 |
| 组织合并 | 将多个组织合并为一个（保留主组织） |
| 迁移员工 | 整建制转移员工到目标部门 |

### 3.2 员工管理 (P0)

### 3.3 架构展示 (P0)

#### 3.3.1 树形视图

- 折叠/展开组织节点
- 显示每个节点的员工数量

- 支持按名称、编码搜索
- 显示组织状态（启用/禁用）

### 3.4 权限控制 (P0)

#### 3.4.1 角色定义

| 角色 | 权限范围 |
|------|----------|
| 系统管理员 | 管理全部组织和员工 |
| 分公司管理员 | 管理本公司及下级组织/员工 |
| 部门管理员 | 管理本部门及下级部门/员工 |
| 普通员工 | 查看所在部门及下级的组织和员工 |

#### 3.4.2 权限规则

- 组织查看权限：用户只能查看自己有权限的组织及其下级
- 员工查看权限：用户只能查看归属自己权限范围内组织的员工
- 编辑权限：仅管理员角色可编辑
- 数据隔离：权限递归检查（ ancestors + descendants ）

### 3.5 审批流集成 (P1)

#### 3.5.1 审批触发场景

| 场景 | 触发条件 |
|------|----------|
| 部门调动 | 员工主部门变更 |
| 组织调整 | 删除/合并/迁移组织 |
| 批量导入 | 一次导入超过 N 人 |

#### 3.5.2 审批链配置

```
审批链配置：
├── 部门内调动（同级）→ 部门负责人审批
├── 跨部门调动 → 原部门负责人 + 新部门负责人
├── 跨分公司调动 → 双方分公司负责人
└── 组织调整 → 调整范围涉及的最高层级负责人
```

### 3.6 HRM 同步 (P2)

#### 3.6.1 同步策略

- **同步方向**：从 HRM 拉取数据（非必须功能）
- **同步频率**：手动触发或定时同步
- **冲突处理**：以 HRM 数据为准，或记录差异供人工确认

## 5. 技术架构

### 5.1 技术栈

| 层级 | 技术选型 |
|------|----------|
| 前端 | React + Ant Design Pro + React Tree |
| 后端 | Python FastAPI |
| 数据库 | MySQL 8.0 |
| 缓存 | Redis（可选） |

### 5.2 层级数据查询优化

采用 **Closure Table** 模式存储组织层级关系：

```sql
-- 组织闭包表
CREATE TABLE org_unit_closure (
    ancestor_id: bigint,
    descendant_id: bigint,
    depth: int
);
```

### 5.3 API 设计

```
组织管理:
GET    /api/v1/org/units              # 获取组织树
GET    /api/v1/org/units/{id}         # 获取组织详情
POST   /api/v1/org/units              # 新增组织
PUT    /api/v1/org/units/{id}         # 编辑组织
DELETE /api/v1/org/units/{id}         # 删除组织
PATCH  /api/v1/org/units/{id}/move    # 移动组织

员工管理:
GET    /api/v1/employees              # 获取员工列表
GET    /api/v1/employees/{id}        # 获取员工详情
POST   /api/v1/employees              # 新增员工
PUT    /api/v1/employees/{id}         # 编辑员工
DELETE /api/v1/employees/{id}         # 删除员工
POST   /api/v1/employees/transfer     # 发起调动
GET    /api/v1/employees/{id}/transfers  # 获取调动历史

导入导出:
POST   /api/v1/employees/import       # 批量导入
GET    /api/v1/employees/export       # 导出员工
```
