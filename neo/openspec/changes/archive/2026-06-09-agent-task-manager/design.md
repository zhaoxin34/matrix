## Context

Agent 任务管理是 Neo 平台的核心功能，用于管理 Agent 执行的各种任务。当前平台已有 Agent Factory（Agent 创建和管理），但缺少任务层面的管理功能。

**当前状态**：
- Agent 可以被创建和配置
- 但没有任务的概念，无法管理任务的执行和追踪

**约束**：
- 需要登录态，但不需要权限控制（任何登录用户都可以管理自己创建的 Agent 的任务）
- 周期任务执行功能暂不实现（需 Agent 模块支持）
- 暂停/继续功能暂不实现（UI 存在，点击显示"暂不支持"）

## Goals / Non-Goals

**Goals:**
- 提供任务列表、详情查看功能
- 支持周期任务的创建、编辑、删除
- 支持任务状态操作（取消、禁用、启用）
- 提供执行记录查看功能
- 支持多维度筛选和搜索

**Non-Goals:**
- Agent 任务执行逻辑（由 Agent 模块负责）
- 暂停/继续功能（暂不实现）
- 执行录像回放（暂不实现）
- 任务分配/派发逻辑
- 权限控制（所有登录用户可访问）

## Decisions

### 1. 数据库设计

**决策**：创建 Task 和 TaskRecord 两张表

**理由**：
- Task 表存储任务元数据（名称、内容、类型、调度配置等）
- TaskRecord 表存储每次执行的记录（结果、过程、时长等）
- 分离设计支持任务的历史追踪

**替代方案**：
- 将执行记录存储在 Task 表的 JSON 字段中（❌ 不支持高效查询）
- 使用单一任务表，每次执行更新状态（❌ 丢失历史记录）

### 2. 任务类型区分

**决策**：通过 `task_type` 字段区分临时任务、周期任务、派发任务，不同类型有不同的操作权限

**理由**：
- 临时/派发任务：只读，不可修改/删除
- 周期任务：可创建、修改、删除（仅限无执行记录）

**替代方案**：
- 不同任务类型使用不同的表（❌ 增加复杂度，不利于统一查询）
- 通过权限控制替代类型区分（❌ 增加权限判断逻辑）

### 3. last_exec_status 字段

**决策**：Task 表使用 `last_exec_status` 而非 `exec_status`

**理由**：
- 任务可能有多个执行记录，`exec_status` 语义不清
- `last_exec_status` 明确表示"上次执行的状态"，便于筛选和展示

### 4. Cron 表达式验证

**决策**：后端验证 Cron 表达式格式

**理由**：
- 前端验证可能被绕过
- 确保存储的表达式格式正确

### 5. 暂停/继续前端处理

**决策**：暂停/继续按钮点击后，前端直接显示"暂不支持"，不调用后端 API

**理由**：
- Agent 模块尚未实现暂停/继续能力
- 预留 API 端点，未来 Agent 实现后可启用
- 避免后端返回"功能未实现"错误

## Risks / Trade-offs

- [Risk] 周期任务创建后不会自动执行
  → [Mitigation] 在创建成功提示中说明"任务调度功能暂未开放"

- [Risk] 用户可能期望修改正在执行的任务
  → [Mitigation] 编辑前检查任务状态，执行中不允许编辑

- [Risk] 删除周期任务后，执行记录仍然存在
  → [Mitigation] 删除前检查是否有执行记录，无记录才允许删除

## Migration Plan

1. 创建数据库表（task, task_record）
2. 实现后端 API（13 个端点）
3. 实现前端 UI（4 个页面）
4. 集成测试
5. 上线部署

**回滚策略**：数据库表使用软删除标记，`status` 字段支持快速禁用
