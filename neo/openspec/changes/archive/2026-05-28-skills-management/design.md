## Context

Neo 项目需要构建 Skills 管理系统，为 Agent 提供标准化的技能支持能力。Skills 是由 markdown 文档和脚本文件组成的多文件复合体，需要支持：
- 文件的多级目录结构管理
- 版本化的文件内容存储（每次修改保留历史）
- 草稿与发布的分离设计
- 完整的生命周期管理（draft → active → disabled）

**当前状态**：已完成产品设计和 API 接口设计文档。

## Goals / Non-Goals

**Goals:**
- 实现 Skills 的 CRUD 操作（创建、读取、更新、软删除）
- 实现文件管理功能（文件树展示、文件内容读写）
- 实现版本管理功能（发布、回滚、版本历史）
- 实现状态机流转（draft → active → disabled）
- 实现基于角色的权限控制

**Non-Goals:**
- 不支持文件的物理删除（保留所有历史版本）
- 不支持多租户隔离（Skills 为全局资源）
- 不支持文件上传下载（仅在线编辑）
- 不支持 Skill 克隆或复制

## Decisions

### Decision 1: 草稿与发布分离设计

**选择**: 使用 `draft_snapshot` JSON 字段存储草稿，`SkillVersion` 表存储发布记录

**理由**:
- 避免每次编辑都创建版本记录，减少数据库写入
- 草稿编辑无需版本号，降低用户心智负担
- 回滚操作简单（快照覆盖），无需复制大量文件

**替代方案考虑**:
- 方案 B: 每次编辑都创建版本记录 → 缺点是版本过多，难以管理
- 方案 C: 使用文件存储服务 → 缺点是引入外部依赖，增加复杂度

### Decision 2: 文件系统存储设计

**选择**: 使用 `FileMetadata + File` 双表设计，通过 `path` 字段隐式表达目录结构

**理由**:
- 目录结构通过 path 字段隐式表达（如 `a/b/c/d.md`），无需单独的 directory 实体
- 每次修改文件在 File 表新增记录，保留完整历史
- file_snapshot 通过 `file_metadata_id + file_id` 锁定历史时刻的文件集合

**替代方案考虑**:
- 方案 B: 使用 JSON 存储目录结构 → 缺点是查询效率低，难以按目录筛选
- 方案 C: 使用对象存储服务 → 缺点是引入外部依赖，增加成本

### Decision 3: API 路径设计

**选择**: RESTful 风格，嵌套路径表示资源从属关系

**理由**:
- `/api/v1/skills` - Skills 集合操作
- `/api/v1/skills/{code}` - 单个 Skill 操作
- `/api/v1/skills/{code}/files` - Skill 的文件集合
- `/api/v1/skills/{code}/files/{path}` - 单个文件操作

**替代方案考虑**:
- 方案 B: 平面化 API 路径 → 缺点是语义不清晰，RESTful 程度低
- 方案 C: GraphQL API → 缺点是增加复杂度，对于本场景收益不明显

### Decision 4: 状态机设计

**选择**: draft → active → disabled，单向流转，不可逆

**理由**:
- 草稿状态可自由编辑
- 发布后状态变为 active，可被调用
- 禁用后状态变为 disabled，不可调用，也不允许再发布

**替代方案考虑**:
- 方案 B: 支持状态回退（如 disabled → active）→ 缺点是增加复杂度，实际场景不需要
- 方案 C: 引入更多状态（如 deprecated）→ 缺点是状态机复杂，当前场景不需要

### Decision 5: 权限模型设计

**选择**: 基于角色的权限控制，通过 `create_user_id` 区分所有者

**理由**:
- super_admin 和 skill_admin 具有全部权限
- skill_editor 只能编辑自己创建的 Skill
- skill_viewer 只能查看 Skill

**替代方案考虑**:
- 方案 B: 基于 Workspace 的权限隔离 → 缺点是 Skills 为全局资源，不适合 Workspace 隔离
- 方案 C: 基于组织的权限控制 → 缺点是增加复杂度，当前场景不需要

## Risks / Trade-offs

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| 文件路径冲突 | 同一 Skill 下文件路径不能重复 | API 层校验 path 唯一性 |
| 版本号冲突 | 同一 Skill 下版本号必须唯一 | API 层校验版本号唯一性 |
| 文件内容过大 | 大文件可能导致 LONGTEXT 字段性能问题 | 限制单文件大小（如 10MB） |
| 草稿快照过大 | JSON 字段存储大量文件引用可能影响性能 | 草稿文件数量限制（如 100 个） |
| 权限模型简化 | 当前模型未区分「编辑自己」vs「编辑他人」 | 未来可扩展为更细粒度的权限控制 |

## Open Questions

2. **草稿文件数量**: 每个 Skill 草稿中文件数量限制多少？（建议 100 个）
3. **已发布文件删除**: 已发布的文件是否允许删除？（建议不允许）
4. **版本号格式**: 是否强制使用语义化版本号（如 1.0.0）？（建议允许任意字符串）
