## 🎯 功能概述

- 指令驱动目标软件：用户选择Agent（注：每个用户拥有多个Agent）,  输入命令，如`填写表单并提交`, Agent Steer将用户所在页面dom结构和指令发送给Agent，Agent把操作Dom的指令返回，Agent Steer执行，期间可能会出问题，比如页面报错、超时、dom匹配错误等等问题，这时需要Harness工程技术，这些细节将在技术文档中描述，非本文档范围
- 任务驱动目标软件：类似指令驱动，只不过`指令`来自定期的Agent Task
- 聊天界面：用户在界面上发送指令的输入界面
  - 技能选择：选择某个技能
  - 会话管理：创建、销毁会话

## ❓ 待讨论问题

### 1. Agent 的驱动来源是什么？

- 用户给一个**目标**（如"帮我把这个表单填完提交"），Agent 自己拆解步骤
- 还是用户给一个**操作序列**（如"点这里→输入这个→点提交"），Agent 只是执行

### 2. Agent 怎么"看懂"页面？

- 页面上有哪些可交互元素（按钮、输入框、下拉等）
- 当前页面处于什么状态（哪些字段已填、哪些校验过了）
- 怎么定位到正确的元素

### 3. 用户在操作过程中的角色是什么？

- **监督**：Agent 操作过程用户全程可见，随时可接管
- **审批**：Agent 提出操作计划，用户确认后再执行
- **旁观**：Agent 自己干，用户只看结果

### 4. 操作失败了怎么办？

- 找不到元素 → 跳过？重试？提示用户？
- 操作被页面拦截 → 怎么感知？怎么恢复？
- 权限不足 → 提示授权？

## 💡 我的理解与回答

### 1. 驱动来源

**前期用操作序列，后期逐步引入目标拆解。**

前期目标：用户给一个明确的操作序列，Agent 按顺序执行。这样技术风险最低，可以先验证"Agent 驱动页面"这件事本身是否可行。

后期演进：随着 Agent 对目标系统的理解加深，可以让 Agent 自己拆解目标、规划步骤。这是一个需要积累的过程，不是一开始就要做到的。

### 2. 怎么"看懂"页面

**用 DOM 分析 + 可访问性语义。**

具体路径：
1. Content Script 收集页面的 DOM 树结构
2. 结合 Accessibility Tree（无障碍树），提取可交互元素的语义信息
3. 给每个元素打上结构化标签：类型（输入/点击/选择）、标识符、当前值、状态
4. 将这些信息传给 Agent Steer（iframe），Agent 基于此决策

rrweb 的录制事件流也可以复用——它本质上记录的就是 DOM 操作事件，可以反推出页面结构。

不依赖截图 + 视觉，因为：
- 视觉方案成本高、准确率不稳定
- DOM + Accessibility 更精确，也更容易结构化

### 3. 用户在操作过程中的角色

**前期：监督模式。**

Agent 操作时，用户全程可见。Agent 的每一步操作都高亮显示，用户随时可以暂停、接管或终止。

原因：
- 信任建立需要过程，先让用户看到 Agent 在干什么
- 出了问题用户可以马上干预，避免造成不可逆影响
- 技术验证阶段，监督模式也方便调试和问题定位

后期可以演进到"审批模式"（Agent 提出计划，用户确认后再执行）。

### 4. 操作失败了怎么办

**分级处理，用户可干预。**

| 失败类型 | 处理策略 |
|---------|---------|
| 找不到元素 | Agent 暂停，提示用户"找不到 XXX，请确认页面状态"，用户可选择跳过或接管 |
| 操作被拦截（弹窗、校验失败等） | Agent 感知到页面变化，暂停，提示用户处理后继续 |
| 权限不足 | 提示用户授权，Agent 等待 |
| 超时 | 重试一次，仍失败则停止并汇报 |

核心原则：**Agent 不要擅自做不可逆决策**，不确定时就暂停等用户确认。
