# 后端审核报告

## 1. 功能理解

### 1.1 产品需求理解

Neo 系统的组织管理模块是层级结构管理系统，包含以下核心功能：

| 模块 | 功能点 | 说明 |
|------|--------|------|
| **组织管理** | 组织 CRUD | 创建、查看、编辑、删除、禁用/启用、移动组织 |
| **组织层级** | 树形结构 | 最多 4 级（公司→部门→子部门→小组） |
| **员工管理** | 员工 CRUD | 创建、查看、编辑、调动员工 |
| **员工与组织** | 多对多关系 | 主属部门 + 辅助部门 |
| **状态机** | 组织状态 | active ↔ inactive |
| **状态机** | 员工状态 | onboarding → on_job → transferring/offboarding |
| **权限控制** | 超级管理员 | 仅超级管理员可访问组织管理功能 |

### 1.2 核心功能总结

1. **组织树形结构**：使用闭包表（Closure Table）存储层级关系
2. **删除/禁用约束**：删除需校验子组织、员工、Workspace；禁用需校验有效 Workspace
3. **员工与用户解耦**：Employee 与 User 独立，通过 Mapping 表关联
4. **直接调动**：无需审批，任何人可发起员工调动

## 2. 数据模型评估

### ✅ 合理的设计

| 表 | 设计 | 说明 |
|----|------|------|
| `organization_unit` | 合理 | 字段完整，包含层级信息 |
| `org_unit_closure` | 合理 | 闭包表设计正确，支持高效层级查询 |
| `employee` | 合理 | 员工核心属性完整 |
| `employee_secondary_unit` | 合理 | 联合唯一约束正确，防止重复分配 |
| `employee_transfer` | 合理 | 调动历史记录完整 |
| `user_employee_mapping` | 合理 | 1:0..1 关系，支持员工暂未绑定用户 |

### ⚠️ 需确认的点

| 问题 | 说明 | 建议 |
|------|------|------|
| `leader_id` 类型 | 技术设计中为 INT，产品设计中为 bigint | 建议统一为 BIGINT，与主键类型一致 |
| 闭包表索引 | 设计中未明确索引 | 建议在 `ancestor_id` 和 `(ancestor_id, descendant_id)` 上建立索引 |
| `primary_unit_id` 可为空 | 技术设计中标注为"否"，但员工可无主属部门 | 建议明确：员工是否必须属于一个主属部门 |

### ❌ 需优化的点

| 问题 | 说明 | 建议 |
|------|------|------|
| 缺少软删除标记 | `organization_unit` 表无 `is_deleted` 字段 | 建议增加软删除字段，便于数据恢复和审计 |
| 缺少审计字段 | 创建人、更新人字段缺失 | 建议增加 `created_by`、`updated_by` 字段 |

## 3. API 设计评估

### ✅ 规范的设计

| 接口 | 评价 |
|------|------|
| `GET /api/v1/org/units` | ✅ RESTful 规范，支持状态过滤 |
| `POST /api/v1/org/units` | ✅ 符合规范 |
| `PUT /api/v1/org/units/{id}` | ✅ 符合规范 |
| `PUT /api/v1/org/units/{id}/move` | ✅ 资源迁移设计合理 |
| `GET /api/v1/employees` | ✅ 支持分页、过滤、搜索 |
| `POST /api/v1/employees` | ✅ 请求体设计合理 |
| `PUT /api/v1/employees/{id}` | ✅ 符合规范 |
| `PUT /api/v1/employees/{id}/transfer` | ✅ 直接调动设计正确 |

### ⚠️ 需确认的接口

| 问题 | 说明 |
|------|------|
| 移动组织接口 | `PUT /api/v1/org/units/{id}/move` 是合适的设计，但技术设计中对移动后层级校验描述不够明确 |
| 批量导入/导出 | 技术设计包含接口，但产品设计标记"暂不做"，需明确状态 |

### ❌ 需要修正的设计

| 问题 | 说明 | 建议 |
|------|------|------|
| 删除员工接口冲突 | 技术设计包含 `DELETE /api/v1/employees/{id}`，但产品设计明确"不提供员工删除功能" | 技术设计应删除此接口，或明确为内部软删除操作 |
| 软删除描述矛盾 | 技术设计说"软删除将 status 设为 offboarding"，但这与产品设计的员工状态机冲突 | 软删除应使用独立的 `is_deleted` 字段，而非改变业务状态 |

### ✅ 响应格式

统一响应格式设计合理：
```json
{
  "code": 0,
  "message": "ok",
  "data": {...},
  "traceId": "abc-123",
  "timestamp": 1713700000000
}
```

## 4. 业务逻辑评估

### ✅ 正确的设计

| 业务逻辑 | 评价 |
|----------|------|
| 组织层级限制 4 级 | ✅ 技术设计应在前端和后端双重校验 |
| 删除约束（子组织、员工、Workspace） | ✅ 校验逻辑完整 |
| 禁用约束（有效 Workspace） | ✅ 校验逻辑清晰 |
| 直接调动（无需审批） | ✅ 设计符合产品需求 |
| 员工状态机 | ✅ 状态转换设计正确 |

### ⚠️ 需补充的校验逻辑

| 校验项 | 说明 |
|--------|------|
| 移动组织后层级校验 | 移动可能导致超过 4 级限制，需在后端校验 |
| 组织名称同级唯一 | 产品设计要求"名称在同一父级下唯一"，需实现 |
| 组织编码全局唯一 | 已有 UNIQUE 约束，校验逻辑需在应用层实现 |
| 员工工号唯一 | 已有 UNIQUE 约束，校验逻辑需在应用层实现 |
| 手机号格式校验 | 需明确正则表达式（国内手机号 11 位） |
| 邮箱格式校验 | 需明确正则表达式 |

### ⚠️ 需明确的状态机细节

| 问题 | 说明 |
|------|------|
| 员工状态转换触发条件 | 技术设计未说明如何触发状态转换（如 on_job 如何从 onboarding 转换而来） |
| 调动后状态 | 是否需要从 on_job 变为 transferring，再到 on_job？还是直接变更主属部门？ |

## 5. 建议与问题

### 5.1 数据模型建议

```sql
-- 建议增加软删除字段
ALTER TABLE organization_unit ADD COLUMN is_deleted TINYINT(1) NOT NULL DEFAULT 0;
ALTER TABLE employee ADD COLUMN is_deleted TINYINT(1) NOT NULL DEFAULT 0;

-- 建议增加审计字段
ALTER TABLE organization_unit ADD COLUMN created_by INT;
ALTER TABLE organization_unit ADD COLUMN updated_by INT;

-- 建议增加闭包表索引
CREATE INDEX idx_closure_ancestor ON org_unit_closure(ancestor_id);
CREATE UNIQUE INDEX idx_closure_ancestor_descendant ON org_unit_closure(ancestor_id, descendant_id);
```

### 5.2 API 建议

| 建议 | 说明 |
|------|------|
| 统一错误码 | 建议定义错误码枚举，如 `ORG_NOT_FOUND = 1001` |
| 补充验证注解 | 使用 Pydantic 的 `Field` 描述约束，如 `Field(..., min_length=1, max_length=100)` |
| 分页默认值 | 建议设置合理的默认值（page=1, page_size=20, max=100） |

### 5.3 潜在问题

| 问题 | 风险等级 | 说明 |
|------|----------|------|
| 产品设计 vs 技术设计不一致 | 🔴 高 | 删除员工接口冲突，需明确 |
| 移动组织层级校验缺失 | 🔴 高 | 可能导致超过 4 级限制 |
| 闭包表维护复杂性 | 🟡 中 | 移动组织时需更新闭包表，可能有性能问题 |
| 员工状态转换逻辑不明确 | 🟡 中 | 需明确状态触发条件和转换路径 |
| 批量导入/导出状态不明确 | 🟡 中 | 技术设计包含但产品设计标记"暂不做" |

## 6. 结论

### 🔴 需要修复后开发

**阻塞问题（需在开发前解决）**：

1. **删除员工接口冲突**：产品设计明确"不提供删除功能"，技术设计包含 `DELETE /api/v1/employees/{id}`，需明确这是内部软删除还是前端可见的删除操作
2. **移动组织层级校验缺失**：需在后端实现移动后的 4 级层级限制校验
3. **组织名称同级唯一校验缺失**：需实现同一父级下名称唯一性校验

**建议改进（非阻塞）**：

1. 统一字段类型（BIGINT）
2. 增加软删除字段
3. 增加审计字段
4. 补充闭包表索引
5. 明确批量导入/导出状态
6. 明确员工状态转换触发条件
